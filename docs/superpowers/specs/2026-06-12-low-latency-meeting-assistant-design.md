# Windows 低延迟会议悬浮助手设计

日期：2026-06-12

## 目标

构建一个 Windows 优先的桌面悬浮助手。它只监听系统输出音频，例如会议中播放到耳机里的对方声音，不采集用户自己的麦克风。从用户点击“开始”到点击“结束”，应用需要完成：

- 近实时显示对方语音字幕。
- 以尽可能低的延迟生成建议回复。
- 保存整轮对话文本和 AI 建议回复。
- 支持将会话导出为 Markdown 或纯文本。

核心产品约束是低延迟。默认回复策略采用“端点触发”，不默认基于不稳定 partial 字幕提前猜测生成。

## 延迟目标

设计优先满足以下实用目标：

- 字幕 partial 文本出现：300-800 ms。
- 字幕 final 文本稳定：对方说完后 800-1800 ms。
- 建议回复首字出现：对方完成一个语义片段后约 800-1500 ms。
- 短建议回复完整输出：1500-3000 ms。
- 长建议回复流式持续输出，并且生成中也可复制。

在 ASR 服务、网络、连接复用、短 prompt 和低 TTFT 模型都优化到位时，可以争取以下拉伸目标：

- 端点后约 600 ms 出现回复首字。
- 短回复约 1500-2500 ms 完成。

## 默认低延迟策略

应用应区分不同置信度的转写结果：

- partial ASR 只用于实时字幕显示。
- final 或 near-final ASR 用于触发 AI 回复生成。
- MVP 不做基于 partial ASR 的提前投机生成。

这样可以避免因为听错、语义未收束、对方话没说完而生成错误回复。

主要通过以下方式压缩固定延迟：

- 静音端点确认窗口初始设为约 300 ms。
- 端点确认采用自适应策略，而不是写死常量。
- 选择 final 结果快的流式 ASR 服务。
- 选择 TTFT 低的流式 LLM 服务。
- 使用长连接、预热、短上下文和短 prompt。

## 音频采集

Windows 音频采集使用 WASAPI loopback，目标是用户选择的输出设备。默认设备是当前系统输出设备，通常是耳机或扬声器。

关键行为：

- 只采集系统输出音频。
- 不打开、不请求、不使用麦克风采集路径。
- 枚举输出设备，并允许用户手动选择。
- 会话中检测设备变化，并尝试自动重连。
- 使用小 buffer，约 10-20 ms 音频帧。
- 将音频转换成 ASR 服务需要的 PCM 格式。

已知限制：如果用户把自己的麦克风监听回放到系统输出，或启用了混音/监听功能，应用可能捕获这部分回放音频。应用需要提示这一点，并提供一个简单的输入/输出检查。

## 端点检测

端点检测是低延迟体验的核心杠杆。

默认端点策略：

- 使用本地能量检测/VAD 跟踪语音活动。
- 初始静音确认窗口为 300 ms。
- 干净短句场景可缩短到 250-300 ms。
- 背景噪声、误切频繁、长停顿或 ASR 状态不确定时放宽到 400-500 ms。

端点检测应作为可调管线组件，而不是硬编码常量。后续 UI 可以暴露简单模式：

- 快速：更低端点延迟，但字幕更可能被切碎。
- 均衡：默认自适应端点策略。
- 稳定：更高端点延迟，减少过早切分。

MVP 内部使用“均衡”策略，暂不暴露复杂参数。

## ASR 管线

ASR 服务必须支持音频流式输入和 interim/partial 结果。

管线：

1. WASAPI 捕获系统输出音频帧。
2. 音频帧进入本地 ring buffer。
3. 通过持久 WebSocket 或等价流式连接发送给 ASR。
4. partial 转写立即更新字幕区。
5. final 转写写入本轮会话日志。
6. final 或 near-final 转写事件触发回复生成。

ASR 结果模型：

- `partial`：不稳定实时文本，仅用于显示。
- `final`：稳定文本，用于保存和生成回复。
- `endpoint`：本地或 ASR 提供的停顿/语义完成信号。

应用需要处理 partial 到 final 的文本修正，避免在会话日志里产生重复字幕。

## 回复生成

默认只基于稳定或接近稳定的上下文生成建议回复。

触发策略：

- 在端点事件加 final/near-final 文本后触发。
- 带入用户可见的最近对话上下文。
- 控制上下文长度，优先保证 TTFT。
- 如果后续 final 文本与当前生成依据差异很大，则取消或替换当前生成。
- 如果只是轻微措辞修正，则继续当前流式生成。

Prompt 策略：

- 系统指令保持短小，聚焦生成可直接说出口的回复。
- 使用滚动窗口保留最近对话。
- 要求回复简洁、明确、可立即使用。
- 实时链路不做重型摘要或检索。

LLM 一旦返回 token，就立即流式显示在悬浮窗中。当前正在生成的建议回复也应该始终可复制。

基于 partial 的投机生成可以作为后续“极速模式”开关，但不进入默认 MVP。

## 悬浮窗

应用提供适合会议场景的 Windows always-on-top 悬浮窗。

MVP 布局：

- 顶部紧凑控制栏：开始、暂停、结束、状态指示。
- 实时字幕区：显示最近 partial/final 字幕。
- 建议回复区：流式显示回复，并提供复制按钮。
- 可折叠会话历史面板。
- 结束会话后提供保存状态和导出入口。

交互要求：

- 窗口可拖动。
- 窗口可折叠成小条。
- 建议回复一键复制。
- 当前监听/暂停/结束状态必须明显。
- UI 更新不能阻塞音频采集或网络流。

第一版应克制、实用，不做复杂仪表盘。

## 会话保存

每次点击“开始”创建一个 session。点击“结束”关闭该 session。

本地存储建议使用 SQLite。

Session 字段：

- `id`
- `title`
- `started_at`
- `ended_at`
- `output_device_id`
- `created_at`

Event 字段：

- `id`
- `session_id`
- `type`：transcript、suggestion、system
- `text`
- `is_final`
- `started_at_ms`
- `ended_at_ms`
- `created_at`

默认只保存文本和元数据。音频保存默认关闭，并且不属于 MVP 范围。

导出格式：

- 带时间戳的 Markdown transcript。
- 纯文本 transcript。

保存和导出都走后台队列，不能阻塞实时链路。

## 架构

推荐第一版技术栈：

- 桌面壳：Tauri。
- 原生音频：Rust WASAPI loopback 模块。
- UI：React 或其他轻量 Tauri 前端。
- 本地数据：SQLite。
- ASR：支持 interim/final 的云端流式 ASR。
- LLM：低 TTFT 的流式模型/服务。

主要模块：

- `audio-capture`：输出设备枚举和 WASAPI loopback 音频帧。
- `audio-stream`：缓冲、重采样和 ASR 流式传输。
- `transcript-engine`：partial/final 合并、修正和端点事件。
- `reply-engine`：触发策略、prompt 构造、流式生成和取消。
- `session-store`：SQLite 写入、会话生命周期和导出。
- `floating-ui`：字幕、回复、控制和历史面板。

线程/任务模型：

- 音频采集运行在高优先级 native worker。
- ASR 流式连接独立于 UI 渲染。
- LLM 流式生成独立运行，并支持取消。
- SQLite 写入通过异步队列执行。
- UI 订阅状态更新，不拥有实时管线。

## MVP 范围

MVP 包含：

- Windows-only 应用。
- 开始、暂停、结束会话控制。
- 从选定输出设备进行 WASAPI loopback 采集。
- 流式 ASR 和实时 partial 字幕。
- 端点触发的流式 AI 回复建议。
- 从开始到结束保存整轮文本。
- Markdown 和纯文本导出。
- 复制当前建议回复。

MVP 不包含：

- 麦克风采集。
- 说话人分离。
- 音频文件录制。
- 本地 Whisper 或本地 LLM 模式。
- 知识库检索。
- Zoom/Teams/Meet 深度平台集成。
- 实时会议总结。
- partial 字幕投机生成回复。

## 风险

音频路由风险：

不同耳机、蓝牙设备、会议软件可能在会话中切换输出设备。应用需要重连逻辑和清晰状态提示。

端点风险：

300 ms 静音窗口在某些自然讲话场景中可能切得太早。自适应端点和 transcript 修正可以降低风险。

网络风险：

流式 ASR 和 LLM 的体验依赖服务延迟。需要持久连接、短 prompt、服务选型和延迟监控。

隐私风险：

会议转写可能很敏感。应用必须明确显示监听状态，默认只本地保存文本，并且 MVP 不保存原始音频。

成本风险：

长会议中的流式 ASR 和 LLM 调用会产生成本。产品后续应统计会话时长和用量，并提供成本控制。

## 验证计划

早期验证应重点测量延迟，而不是只验证功能是否可用。

测试场景：

- 本地音频播放到耳机。
- Teams/Zoom/浏览器会议音频。
- 蓝牙耳机设备切换。
- 带 200-500 ms 停顿的短句。
- 背景噪声和长静音。
- 至少 60 分钟长会话。

指标：

- 音频帧采集延迟。
- ASR partial 延迟。
- ASR final 延迟。
- 端点确认延迟。
- LLM TTFT。
- 建议回复完成时间。
- UI 渲染延迟。
- 会话写入队列滞后。

MVP 验收标准：

- 不使用麦克风权限或麦克风采集路径。
- 对方说话时能出现 partial 字幕。
- 默认只在端点/final 或 near-final 后触发回复生成。
- 正常网络下，建议回复首字进入目标延迟范围。
- 结束会话后能保存完整文本对话。
- Markdown 导出包含会话时间线和建议回复。
